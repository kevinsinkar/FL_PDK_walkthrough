FlexLogger manages your plug-in through a state machine with five states. Each state maps to a specific VI in your LabVIEW project. The lifecycle is not linear — Cleanup Session runs every time the user changes a setting, looping back up to Configure Session, and only routes forward to Finalize when the plug-in is removed.

ONCE Step 2.1 — Initialize

Runs once when the plug-in is added to a FlexLogger project. This is where you declare your channels, parameters, and timing method. No hardware interaction here.

🕒Periodic / ImmediateFor Producers & Consumers
🌊ResamplingFor Calculators
⚠ Choose one: Set Plugin Timing or Set Plugin Resampling — typically not both. Calculators that read input waveforms should use resampling.
Initialize.vi

REPEAT Step 2.2 — Configure Session

Runs every time the user changes a setting (while the test is stopped), and once on initial setup. Open hardware connections, read parameter values, and set stream timing (dt, T0) here. Store hardware references in your class private data.

Configure Session.vi

LOOP Step 2.3 — Process (Core Loop)

The main loop. FlexLogger calls this VI repeatedly at the timing rate you configured. Read from hardware, perform calculations, and write data to FlexLogger here. No while-loop needed — FlexLogger handles the looping.

Running at configured interval (e.g., every 100 ms)
✔ Always check Get Valid Channels before writing. Disabled channels still exist in the array but must not receive data.
Process.vi

REPEAT Step 2.4 — Cleanup Session

Runs when the user makes a configuration change, and again before Finalize when the plug-in is removed. Close hardware connections and release every resource that Configure Session opened. This is the mirror of Configure Session.

Cleanup Session.vi
↻ On config change: Cleanup Session loops back up to Configure Session — Finalize is skipped

ONCE Step 2.5 — Finalize

Runs once when the plug-in is removed from the project or the project closes. Reached only via the removal branch from Cleanup Session. Cleans up anything created in Initialize. Usually empty.

Finalize.vi

User enters config values

Opens the hardware connection using values from the front panel. Runs once before the main loop.

One-time setup before loop

Reads hardware, processes data, writes to chart. Repeats until user clicks Stop.

Core logic repeats each iteration

Runs once after loop exits

For VIs that allocate persistent resources (refnums, queues, DVRs), the absolute last step before the VI fully unloads.

FlexLogger Plug-In Structure
Initialize.vi
Step 3.1 — Declare Parameters & Channels

Use Write Parameter and Create Channel to define what appears in FlexLogger's UI. Set the timing method.

Read parameter values from FlexLogger, open instrument sessions, set stream timing. Store references in MyPlugin.ctl.

Follow these five steps to go from a fresh install to a working plug-in loaded in FlexLogger. Each step builds on the previous one.

1

Step 4.1 — Install the Tools

1a
Install LabVIEW (with Application Builder Module)
1b
Install FlexLogger (check version compatibility first)
1c
Install the FlexLogger Plug-In Development Kit via NI Package Manager
⚠ Install in this exact order — LabVIEW → FlexLogger → PDK

Verify your install: open LabVIEW → View → Functions Palette → FlexLogger. If you see the FlexLogger palette, you're ready.

2

Step 4.2 — Create Your Plug-In Project

2a
In LabVIEW: File → Create Project → FlexLogger IO Plug-in
2b
Choose your template: Produce Data, Perform Calculation, or Consume Setpoint (see Tab 1 for guidance)
2c
Name your plug-in and click Create

The wizard creates: a .lvclass (with all five VIs), an .xml metadata file, and a Packed Library build specification.

💡 Tip from the R&D team: If you plan to integrate real hardware, first build a standalone LabVIEW VI that communicates with your device. Get that working, then map the code into the plug-in VIs. The Mouse Input example demonstrates this pattern.
3

Step 4.3 — Customize Your VIs & Build

3a
Edit Initialize.vi: define parameters, channels, timing
3b
Edit Configure Session.vi: read params, open hardware, set stream timing
3c
Edit Process.vi: implement your core logic (read → calculate → write)
3d
Edit Cleanup Session.vi: close hardware, release resources
3e
Edit Finalize.vi: clean up anything created in Initialize (usually empty)
3f
Right-click the Packed Library build spec → Build
Output: MyPlugin.lvlibp + MyPlugin.xml

Build output auto-deploys to: %public%\Documents\National Instruments\FlexLogger\Plugins\IOPlugins\

📝 Versioning: After each rebuild, update the <Version> tag in your XML file to track changes. FlexLogger uses this to detect updates.
4

Step 4.4 — Test with the Simulator

4a
Open Plugin Environment Simulator.vi (ships with the PDK)
4b
Run it and select your .lvclass file
4c
Set breakpoints, add probes, configure parameters, verify output data

The simulator lets you debug from LabVIEW source code — something you can't do once the plug-in is compiled and running inside FlexLogger.

⚠ The simulator does not support digital channels
5

Step 4.5 — Load in FlexLogger & Validate

5a
Launch FlexLogger → open or create a project
5b
Add channels → Plug-in → Select your plug-in
5c
Configure parameters → run test → verify live data on screen

⚠ Critical: File Naming Must Match

📂 MyPlugin/ (folder)
  ├── MyPlugin.lvlibp ✔
  └── MyPlugin.xml ✔

Error state:
  └── DifferentName.xml ✘ Won't load

Per PDF page 22, the folder name, .lvlibp filename, and .xml filename must all match. If they don't, FlexLogger may not show the plug-in in the channel picker.

🔍 Troubleshooting — two failure modes: (1) Folder/file name mismatch: the plug-in simply doesn't appear under Add channels → Plug-in. Verify all three names match. (2) XML parse failure: per PDF page 52, FlexLogger writes an Error.txt file inside the plug-in folder describing the parse problem. Check there first if the XML is suspect.
⚠ You must close & reopen your FlexLogger project to pick up modified plug-ins
⚠ FlexLogger does not support multiple plug-ins with the same name

These are the most common mistakes and gotchas encountered by plug-in developers — sourced from the NI R&D team and the official documentation. Save yourself debugging time by reviewing these before you start.

🔴 Error handling defaults can cause repeated reconfiguration attempts

When a plug-in encounters an error during Configure Session (e.g., hardware not present), FlexLogger's base error handling can trigger Cleanup → Configure cycles that retry the failed setup. If the underlying issue persists, this can manifest as repeated reconfiguration attempts.

✅ Mitigation: Override the error handling behavior so that, on Configure errors, the plug-in stops trying to reconfigure and waits for the user to change a setting. This guidance comes from the FlexLogger Plug-In Development Kit API Reference (linked from PDF page 4), not the User Manual itself — verify the exact VI name and modes against your installed PDK version.

🔴 Calling Add Channels multiple times in the same VI

If you call Add Channels more than once from Initialize.vi (or any single VI), parameter settings won't save correctly. The second call overwrites the first.

✅ Fix: Build an array of all your channels first, then call Add Channels once with the complete array.

🔴 File naming mismatch prevents plug-in from loading

Per PDF page 22, the folder name, .lvlibp filename, and .xml filename must all match. If they don't, the plug-in won't appear in Add channels → Plug-in.

✅ Fix: After building, verify all three names match. Two distinct failure modes to check: (1) name mismatch — plug-in just won't appear; (2) XML parse failure — per PDF page 52, FlexLogger writes Error.txt inside the plug-in folder with details.

🟡 Using both Set Plugin Timing and Set Plugin Resampling

These are mutually exclusive for most use cases. Using both can cause unpredictable Process.vi execution timing.

✅ Fix: Producers → use Set Plugin Timing (Periodic). Calculators → use Set Plugin Resampling. Only combine them in rare cases where timing is independent of input data.

🟡 Calculators that read FlexLogger channels should use Resampling, not Timing

Per PDF page 33, plug-ins that read waveform data from other FlexLogger channels should set Resampling Parameters (Set Plugin Resampling) rather than Timing Parameters (Set Plugin Timing). Resampling triggers Process.vi when time-aligned input data is ready; periodic timing fires on a fixed clock regardless of input availability.

✅ Fix: In Initialize.vi for calculators, use Set Plugin Resampling with the appropriate mode (Minimum, Maximum, Custom, or Original Rate). Set Drift Tolerance and Block Size to control how the resampler waits for aligned data.

🟡 Writing data to disabled channels

Users can disable individual channels in FlexLogger. If your Process.vi writes data to a disabled channel, it can cause errors or unexpected behavior.

✅ Fix: Call Get Valid Channels (or Get Valid Producer Channels) at the start of Process.vi and only write to channels in that list.

🟡 Timestamp overlaps when writing waveforms

Writing waveforms that overlap in time disrupts both visualization and logging. This typically happens when dt or T0 is configured incorrectly.

✅ Fix: Use Set Stream Timing.vi in Configure Session.vi to properly set dt and start time. The Produce Data template shows the correct pattern.

🟡 Sharing data between Handle Notification and Process.vi

Handle Notification.vi runs in a separate thread. Directly modifying class private data from it causes race conditions with Process.vi.

✅ Fix: Use a Data Value Reference (DVR) to safely share state between the notification handler and Process.vi.

🔵 Resampling mode affects output channel rate

When using Maximum resampling mode, all input channels are upsampled to match the fastest channel. Your output channel will run at this faster rate — even if one of the inputs was originally slower.

✅ Tip: This doesn't affect the original channel's logged data, only your calculation output. Use "Produced by Plugin (Resampler timing)" for output channels so they auto-adapt.

🔵 Plug-in changes require FlexLogger restart

After rebuilding a plug-in, FlexLogger doesn't automatically detect the update. You must close and reopen the project (or restart FlexLogger) to load the new version.

✅ Tip: During development, keep FlexLogger closed while rebuilding.
